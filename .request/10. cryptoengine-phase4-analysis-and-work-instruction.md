## 2. 추가 백테스트가 필요한가?

### 결론: **아니오. 백테스트는 충분합니다.**

이유는 다음과 같습니다:

**1) 커버리지가 완전합니다**
- 개별 전략 성능, 레짐 판별, 가중치 최적화, Walk-Forward, 민감도, 스트레스 테스트까지 16개 테스트를 6년 데이터로 수행했습니다.
- 테스트되지 않은 영역이 사실상 없습니다.

**2) FA 전략의 구조적 우위가 반복 확인되었습니다**
- 3년 WF Sharpe 2.434, 6년 단독 Sharpe 1.494
- 스트레스 테스트 4/4 통과, MDD 일관되게 < 1%
- 델타 뉴트럴 구조 덕분에 시장 방향성과 무관한 수익

**3) 추가 테스트의 한계 수익이 극히 낮습니다**
- max_hold 240 A/B 테스트는 Phase 4 실전에서 검증하는 것이 더 유의미
- DCA 재설계는 Phase 4 병행 과제로 이미 결정됨
- 더 많은 백테스트는 과적합 위험만 높입니다

**4) 실제 블로커는 코드 품질입니다**
- 코드 리뷰(#9)에서 발견된 6개 치명 결함이 미수정
- 이 결함들이 해결되지 않으면 백테스트 결과가 아무리 좋아도 Phase 4가 정상 동작하지 않습니다

## 작업 A1: Redis 채널명 통일 (최우선)

### 문제
오케스트레이터가 `strategy:funding_arb:command`로 발행하지만, BaseStrategy는 `strategy:command:funding-arb`를 구독한다. 전략이 오케스트레이터 명령(자본 배분, 시작/정지)을 수신하지 못한다.

### 수정 방법

1. `shared/events.py` 파일을 신규 생성한다:
```python
"""Redis 채널명 상수 — 모든 서비스가 이 파일만 참조한다."""

class Channel:
    FUNDING_RATE   = "market:funding_rate"
    REGIME         = "market:regime"
    ORDER_REQUEST  = "order:request"
    ORDER_RESULT   = "order:result"
    KILL_SWITCH    = "system:kill_switch"

    @staticmethod
    def strategy_command(strategy_id: str) -> str:
        """표준 채널 형식: strategy:command:{strategy_id}"""
        return f"strategy:command:{strategy_id}"

    @staticmethod
    def order_result_for(strategy_id: str) -> str:
        return f"order:result:{strategy_id}"
```

2. `services/orchestrator/core.py`에서 채널 발행 부분을 찾아 `Channel.strategy_command(strategy_id)` 사용으로 변경한다.

3. `services/strategies/base_strategy.py`에서 채널 구독 부분을 찾아 동일한 `Channel.strategy_command(self.strategy_id)` 사용으로 변경한다.

4. `strategy_id` 값 자체도 통일한다: 하이픈(`funding-arb`)과 언더스코어(`funding_arb`) 혼용을 확인하고 하나로 통일한다. (코드 전체 검색 필요)

### 검증
```bash
# 오케스트레이터 → 전략 명령 전달 확인
docker compose up -d strategy-orchestrator funding-arb
docker compose logs -f funding-arb | grep "command"
# "Received command" 로그가 보이면 성공
```

## 작업 A2: set_leverage() + set_margin_mode() 구현

### 문제
거래소에 레버리지 설정 API를 호출하지 않아서, Bybit 기본값(최대 100배)으로 주문이 실행될 수 있다. 격리마진 설정도 누락되어 교차마진으로 전체 계좌가 청산될 위험이 있다.

### 수정 방법

1. `shared/exchange/bybit.py`에 두 메서드를 추가한다:
```python
MAX_LEVERAGE = 2  # 프로젝트 정책 — 절대 이 값을 올리지 않는다

async def set_leverage(self, symbol: str, leverage: int) -> None:
    safe_lev = min(leverage, MAX_LEVERAGE)
    try:
        await self._exchange.set_leverage(safe_lev, symbol)
        log.info("leverage_set", symbol=symbol, leverage=safe_lev)
    except Exception as e:
        if "leverage not modified" in str(e).lower():
            pass  # 이미 설정된 경우
        else:
            raise

async def set_margin_mode(self, symbol: str, mode: str = "isolated") -> None:
    try:
        await self._exchange.set_margin_mode(mode, symbol)
        log.info("margin_mode_set", symbol=symbol, mode=mode)
    except Exception as e:
        if "already" in str(e).lower():
            pass  # 이미 isolated인 경우
        else:
            raise
```

2. `shared/exchange/base.py`(ABC)에 추상 메서드로 추가하여 모든 커넥터가 구현하도록 강제한다.

3. `services/strategies/funding-arb/strategy.py`의 `on_start()`에 호출을 추가한다:
```python
async def on_start(self, capital, params):
    # 기존 코드 위에 추가
    await self._exchange.set_margin_mode(self.symbol, "isolated")
    await self._exchange.set_leverage(self.symbol, 2)
    # ... 기존 on_start 로직
```

### 검증
```bash
docker compose up -d --build funding-arb
docker compose logs funding-arb | grep "leverage_set\|margin_mode_set"
# 두 로그가 모두 보이면 성공
```

## 작업 A3: FA 주문을 실행엔진 경유로 변경

### 문제
`FundingArbStrategy._place_and_track()`이 `self._exchange.place_order()`를 직접 호출하여 SafetyGuard의 6단계 안전 검증(레버리지, 마진, 슬리피지, 네트워크, Rate Limit)을 전부 우회한다.

### 수정 방법

1. `services/strategies/funding-arb/strategy.py`에서 `_place_and_track()` 메서드를 찾는다.

2. `self._exchange.place_order(order)` 호출을 `self.submit_order(order)`로 변경한다.
   - `submit_order()`는 `BaseStrategy`에 이미 구현되어 있으며, Redis `order:request` 채널을 통해 execution-engine으로 주문을 위임한다.

3. 체결 확인 로직도 변경한다:
   - 기존: 직접 호출의 반환값으로 체결 확인
   - 변경: `order:result:{strategy_id}` 채널 구독으로 비동기 수신

4. 동일한 패턴이 다른 전략(grid-trading, adaptive-dca)에도 있는지 확인하고 동일하게 수정한다.

### 주의사항
- FA의 양쪽 레그(현물+선물) 동시 주문 로직이 비동기 결과 수신으로 바뀌면서 one-side-fill 복구 로직에 영향이 있을 수 있다. 복구 타이머(3분)가 정상 동작하는지 확인한다.

### 검증
```bash
docker compose up -d --build funding-arb execution-engine
docker compose logs execution-engine | grep "safety_check\|order_received"
# funding-arb의 주문이 execution-engine 로그에 나타나면 성공
```
