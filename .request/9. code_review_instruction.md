# 비트코인 선물 자동매매봇 코드 리뷰 지시서

> **목적**: 현재 프로젝트의 코드베이스를 전설적 트레이더 원칙 및 프로덕션급 봇 아키텍처 기준으로 전면 검토하고, 누락된 기능·개선점·위험 요소를 식별하여 구체적인 개선안을 제시한다.

---

## 검토 방법

1. 먼저 프로젝트 전체 디렉토리 구조를 파악한다
2. 아래 10개 검토 영역을 순서대로 점검한다
3. 각 영역마다 **현재 상태 → 문제점 → 개선안** 형식으로 정리한다
4. 최종적으로 우선순위별 액션 아이템 목록을 제시한다

---

## 영역 1: 아키텍처 및 프로젝트 구조

### 점검 항목

- 이벤트 기반(Event-Driven) 아키텍처를 사용하고 있는가, 아니면 단순 폴링(Polling) 방식인가
- EventBus 또는 유사한 메시지 라우팅 시스템이 존재하는가
- 관심사 분리가 되어 있는가 (아래 패키지 구조 참조)
  - `core/` — 이벤트 버스, 엔진, 데이터 모델
  - `exchanges/` — 거래소 어댑터 (ABC 기반 추상화)
  - `strategies/` — 전략 패턴 (ABC 기반 추상화)
  - `risk/` — 리스크 엔진 (포지션 사이저, 서킷 브레이커, 손실 한도)
  - `execution/` — 주문 실행기, 슬리피지 추정기, TWAP
  - `data/` — 데이터 피드 핸들러, DB 저장소
  - `monitoring/` — 텔레그램 알림, 메트릭 수집
  - `backtesting/` — 백테스팅 엔진, 시뮬레이터
- 새 거래소 추가 시 어댑터 클래스 하나만 추가하면 되는 구조인가
- 새 전략 추가 시 전략 클래스 하나만 추가하면 되는 구조인가
- 전략 로직과 주문 실행 로직이 분리되어 있는가

### 핵심 기준

> **전설적 트레이더 원칙 #9 (프로세스와 시스템 > 예측)**: 시스템의 각 구성 요소가 독립적으로 테스트·교체 가능해야 한다. 전략을 바꿔도 리스크 엔진은 불변이어야 한다.

## 영역 2: 리스크 관리 엔진

### 점검 항목

- **모든 주문이 리스크 엔진을 반드시 통과하는가** (우회 경로가 없는가)
- 거래당 리스크 한도가 구현되어 있는가 (권장: 자본의 1~2%)
- 포지션 사이징 알고리즘이 존재하는가
  - 고정 비율(Fixed Fractional) 방식인가
  - Half-Kelly 기준이 적용되어 있는가
  - ATR 기반 변동성 조정 사이징이 구현되어 있는가
  - 공식: `포지션_크기 = 계좌_리스크 / (ATR × 승수)`
- 포트폴리오 히트(총 오픈 리스크)를 계산·제한하는가 (권장: 6~10%)
- 상관관계 기반 포지션 크기 조정이 있는가 (BTC/ETH 동시 보유 시 축소)
- 드로다운 기반 서킷 브레이커가 구현되어 있는가

  ```
  0~20% 드로다운  → 정상 매매 (1.0배 크기)
  20~30% 드로다운 → 축소 매매 (0.5배 크기)
  30~50% 드로다운 → 최소 매매 (0.1배, 최고 확신만)
  50%+ 드로다운   → 매매 전면 중단 (0.0배)
  ```

- 일일/주간/월간 손실 한도가 구현되어 있는가
  - 일일: 2~3%
  - 주간: 5~7%
  - 월간: 10~15%
- 최고 잔고(peak balance)를 추적하고 모든 잔고 변경 시 드로다운을 재계산하는가

### 핵심 기준

> **전설적 트레이더 원칙 #1 (리스크 관리 > 적중률)**: 워뇨띠는 "습득해야 할 기술은 진입 시점이 아니라 리스크 관리"라고 했다. 리스크 엔진은 기능이 아니라 아키텍처의 핵심이어야 한다.

> **전설적 트레이더 원칙 #6 (빠른 손절)**: 유진의 드로다운 프레임워크가 상태 머신으로 구현되어야 한다. 50% 이상 드로다운에서 자동으로 매매가 중단되는가?

## 영역 3: 레버리지 및 마진 관리

### 점검 항목

- 레버리지 하드캡이 코드에 존재하는가 (권장: `MAX_LEVERAGE = 5`)
- 설정 파일이나 환경 변수로 레버리지 상한을 우회할 수 있는 경로가 없는가
- 모든 주문 전에 `exchange.set_leverage()`를 호출하는가
- `create_order()` 내부의 레버리지 파라미터에 의존하지 않는가 (바이낸스에서 무시됨)
- `fetch_positions()`로 실제 적용된 레버리지를 검증하는가
- 마진 모드가 격리(isolated)로 설정되어 있는가 (교차 마진은 치명적 청산 리스크)
- 포지션 개설 전에 `exchange.set_margin_mode('isolated', symbol)`을 호출하는가

### 핵심 기준

> **전설적 트레이더 원칙 #2 (낮은 레버리지)**: 워뇨띠 1.5~2배, 린넨 2배 이하, 데겐 스파르탄 1.1배. 모든 전설적 트레이더가 2~5배를 넘지 않는다. 코드에서 이것이 오버라이드 불가능한 제약으로 존재하는가?

## 영역 4: 매매 전략 및 신호 생성

### 점검 항목

- 현재 구현된 전략은 무엇인가
- 복수의 비상관 전략이 조합되어 있는가 (권장 조합)
  - 그리드 매매 (횡보장)
  - 펀딩비 차익거래 (델타 중립)
  - 평균 회귀 (저점 매수)
  - 추세 추종 (방향성)
  - 통계적 차익거래 (페어 트레이딩)
- 멀티 타임프레임 분석이 구현되어 있는가 (1:4 또는 1:6 비율)
- 확신 점수(Conviction Score) 시스템이 존재하는가
  - 일봉 추세 (+3), 4시간 추세 (+2), 1시간 신호 (+1), 펀딩비 역발상 (+1), OI 다이버전스 (+1)
  - 최소 점수 기준이 있는가 (권장: 8~10점 만점 중 5점 이상)
- 역발상 신호가 구현되어 있는가
  - 펀딩비 >+0.05%/8시간 시 숏 신호
  - 공포·탐욕 지수 <25 시 롱 신호
  - 과도한 오픈 인터레스트 다이버전스 감지
- 최소 R:R 비율 필터가 구현되어 있는가 (권장: 2:1 이상, 이상적: 3:1)
  - `abs(이익실현 - 진입) / abs(진입 - 손절) >= 2.0`
  - 기준 미달 신호 자동 거부 로직이 있는가
- 과잉매매 방지 로직이 구현되어 있는가
  - 일일 최대 매매 횟수 제한 (권장: 3~5회)
  - 매매 간 최소 시간 간격 (권장: 4시간)
  - 연속 손실 후 쿨다운 기간

### 핵심 기준

> **전설적 트레이더 원칙 #3 (역발상 사고)**: GCR의 "군중에 반대하라". 극단적 센티먼트에서 반대 방향 신호를 생성하는 로직이 있는가?

> **전설적 트레이더 원칙 #5 (인내심/과잉매매 방지)**: 프로 스캘퍼 "톰"은 "매매를 적게 할수록 수익성이 높아진다"고 했다.

> **전설적 트레이더 원칙 #10 (비대칭 베팅)**: 피터 브랜트의 "1달러를 4달러로 바꿀 기회만 찾는다". R:R 3:1 미만은 거부하는가?

## 영역 5: 주문 실행 및 에러 처리

### 점검 항목

- 지원하는 주문 유형
  - 시장가(Market)
  - 지정가(Limit)
  - 스탑 로스(Stop-Loss)
  - 트레일링 스탑(Trailing Stop)
  - OCO (이익실현 + 손절 동시 관리)
- ATR 기반 손절가 계산이 구현되어 있는가 (권장: 2× ATR)
- 분할 청산이 구현되어 있는가 (권장: 50/30/20 패턴)
  - 1차 목표: 50% 청산 + 손절을 본전으로 이동
  - 2차 목표: 30% 청산
  - 나머지 20%: 트레일링 스탑
- 대형 주문을 위한 TWAP 실행이 구현되어 있는가
- 슬리피지 추정 및 관리 로직이 있는가
- 부분 체결(Partial Fill) 처리가 되어 있는가
- 재시도 로직이 구현되어 있는가
  - 일시적 오류에만 재시도: `NetworkError`, `ExchangeNotAvailable`, `RequestTimeout`, `RateLimitExceeded`
  - 클라이언트 오류는 즉시 실패: `AuthenticationError`, `InsufficientFunds`, `InvalidOrder`
  - 지수 백오프(Exponential Backoff) 적용
  - `tenacity` 라이브러리 사용 여부
- 멱등성을 위한 `clientOrderId`(UUID) 사용 여부
- 거래 비용 모델링이 현실적인가
  - 왕복 비용: 0.3~0.5% (수수료 + 슬리피지 + 펀딩비)

### 핵심 기준

> 전설적 트레이더들의 분할 진입/청산 방식이 코드에 반영되어야 한다. 단일 진입-단일 청산 구조는 비대칭 수익 구조를 만들 수 없다.

## 영역 6: 데이터 수집 및 저장

### 점검 항목

- 데이터 수집 방식: REST 폴링인가 WebSocket 스트리밍인가
- CCXT Pro의 비동기 WebSocket을 활용하는가
- 멀티 심볼/멀티 거래소 동시 구독이 가능한가 (`asyncio.gather`)
- 데이터베이스 선택
  - TimescaleDB (권장) — 시계열 최적화, 연속 집계
  - PostgreSQL — 차선
  - SQLite — 프로토타이핑에만 적합
- OHLCV 데이터의 다중 타임프레임 저장/조회가 가능한가
- 펀딩비, 오픈 인터레스트, 청산 데이터를 수집·저장하는가
- 수치 정밀도 문제가 있는가
  - CCXT의 float → 내부에서 `Decimal` 사용 여부
  - 17.89999999998 같은 가격으로 주문이 나가지 않는가
- 데이터 무결성 검증 (빈 캔들, 중복, 시간 순서 오류 감지)

### 핵심 기준

> **전설적 트레이더 원칙 #8 (24/7 모니터링)**: 크립토 시장은 365일 24시간 운영된다. WebSocket 연결이 끊어졌을 때 자동 재연결 및 누락 데이터 복구가 되는가?

## 영역 7: 백테스팅 및 전략 검증

### 점검 항목

- 백테스팅 프레임워크가 존재하는가 (VectorBT, Jesse, Backtrader, 자체 구현)
- 워크포워드 최적화(Walk-Forward Optimization)가 구현되어 있는가
  - 훈련 윈도우: 180~365일
  - 테스트 윈도우: 30~90일
  - 최적화 대상: 샤프 비율 또는 소르티노 비율 (원시 수익이 아닌)
- 몬테카를로 시뮬레이션이 구현되어 있는가
  - 거래 순서 셔플 × 10,000회
  - 5번째 백분위수 최대 드로다운 계산
- 백테스팅에서 현실적 비용이 모델링되는가
  - 메이커/테이커 수수료
  - 슬리피지 (과거 스프레드의 2배 모델링)
  - 펀딩비
- 과적합 방지 대책
  - 핵심 파라미터 5개 이하
  - 파라미터 민감도 분석 (±10~20% 변동 시에도 수익성 유지)
  - 최소 100회 이상 거래로 통계적 유의성 확보
  - Look-ahead 편향 방지 (다음 캔들 시가에서 실행)
  - 생존자 편향 방지 (상장 폐지 종목 포함 데이터)
- BTC 단순 보유(Buy & Hold) 대비 성과 비교 기능이 있는가
- 성과 지표 산출이 가능한가
  - 샤프 비율 (최소 > 0.5, 양호 > 1.0)
  - 소르티노 비율
  - 최대 드로다운 (최소 < 30%)
  - 칼마 비율
  - 프로핏 팩터 (최소 > 1.3)
  - 평균 R-배수
- 페이퍼 트레이딩(Sandbox) 모드가 있는가
  - `exchange.set_sandbox_mode(True)` 활용
  - 1~3개월 검증 기간

### 핵심 기준

> **전설적 트레이더 원칙 #9 (프로세스 > 예측)**: 라이트는 "성공적인 포커 플레이어는 결과가 아닌 과정을 최적화한다"고 했다. WFO를 통과하지 않은 전략은 절대 라이브에 투입하지 않는 정책이 코드에서 강제되는가?

## 영역 8: 인프라 및 배포

### 점검 항목

- 배포 환경
  - VPS/클라우드 사용 여부 (권장: AWS 도쿄 리전 ap-northeast-1)
  - 거래소 서버와의 지연 시간을 고려한 리전 선택
- Docker/Docker Compose 컨테이너화 여부
  - `restart: always` 설정
  - 서비스 구성: 봇, DB, Redis, Prometheus, Grafana
- 프로세스 관리
  - systemd 서비스 등록 여부
  - `Restart=always`, `RestartSec=15` 설정
  - 무한 크래시 루프 방지 (`StartLimitBurst=5`)
- 트레이딩 프로세스와 모니터링 프로세스가 분리되어 있는가
- 장애 복구(Failover) 메커니즘
  - 프로세스 크래시 시 자동 재시작
  - WebSocket 재연결 로직
  - 거래소 점검 시 대응 로직
- 로깅
  - 구조화된 로깅 (`structlog`, JSON 출력)
  - 로그 로테이션 (30일 보관)
  - 기록 항목: 주문 제출, 체결, 취소, API 오류, WebSocket 이벤트, 서킷 브레이커 활성화

### 핵심 기준

> **전설적 트레이더 원칙 #8 (24/7 모니터링)**: GCR은 "3년 반 동안 단 하루도 쉬지 않았다"고 했다. 봇이 이 수준의 가동 시간을 보장하는가?

## 영역 9: 모니터링 및 알림

### 점검 항목

- 텔레그램/디스코드 알림 통합 여부
  - 매매 알림 (진입/청산 + PnL)
  - 일일 요약 리포트
  - 오류 알림
  - 하트비트 메시지 (15~30분 간격)
- 원격 제어 명령 구현 여부
  - `/status` — 현재 포지션 조회
  - `/pause` — 매매 중지
  - `/resume` — 매매 재개
  - `/close_all` — 비상 전체 청산
- Prometheus + Grafana 모니터링 대시보드

  - WebSocket 연결 해제 감지

### 핵심 기준
