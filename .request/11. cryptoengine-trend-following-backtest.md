```
[교훈 2] 롱 전용은 하락장에서 구조적 손실 불가피
  → DCA baseline: 2022년 폭락 -65.89%

  → 2022년 하락장에서 숏 수익이 발생하는지 반드시 확인

[교훈 3] 진입 조건 강화 ↔ 거래 빈도 트레이드오프의 딜레마
  → DCA: 조건 강화 → MDD↓ 동시에 거래수↓ → WF 불통과
  → 이번 설계: 진입 조건은 단순하게, 리스크 관리(ATR 스탑)로 MDD 통제
  → "진입 정확도"보다 "손절 규율"이 추세추종의 핵심
```

## Stage 0: FA 수익 재투자 시뮬레이션

### 목적

FA가 펀딩비를 수취할 때마다 **수익의 일부를 BTC 현물 매수**에 자동 배분한다.
원금 리스크 0(FA 수익분만 사용), Phase 4에서 즉시 적용 가능.

### 0-1. 시뮬레이션 로직

```python
class FA_Profit_Reinvest:
    """
    기존 FA short_hold 백테스트 결과에서 수익 발생 시점을 추출하고,
    해당 시점에 수익의 N%를 BTC 시장가 매수했다고 가정한다.
    
    입력:
      - FA 백테스트 거래 내역 (이미 DB에 존재)
      - 1h OHLCV (매수 시점 가격 결정)
    
    파라미터:
      reinvest_ratio: [0.20, 0.30, 0.50, 0.70, 1.00]  # 5가지 비율 테스트
      max_spot_allocation_pct: 20.0  # 포트폴리오 대비 현물 최대 비중
      sell_on_drawdown_pct: None     # 현물은 매도 안 함 (장기 보유)
    
    측정:
      - 6년간 축적된 BTC 현물 수량
      - 현물 포지션의 평균 매입 단가 vs 최종 BTC 가격
      - 포트폴리오 전체 가치 (FA 수익 + 현물 가치)
      - FA 단독 대비 추가 수익률
      - 현물 포지션의 MDD (2022년 폭락 구간)
    """
    
    def simulate(self, fa_trades: list, ohlcv: pd.DataFrame, reinvest_ratio: float):
        spot_btc = 0.0
        spot_cost_basis = 0.0
        total_reinvested = 0.0
        
        for trade in fa_trades:
            if trade.pnl > 0:  # FA 수익이 양수인 거래만
                reinvest_amount = trade.pnl * reinvest_ratio
                btc_price = ohlcv.loc[trade.timestamp, "close"]
                btc_qty = reinvest_amount / btc_price
                
                spot_btc += btc_qty
                total_reinvested += reinvest_amount
                spot_cost_basis = total_reinvested / spot_btc if spot_btc > 0 else 0
        
        # 최종 평가
        final_btc_price = ohlcv.iloc[-1]["close"]
        spot_value = spot_btc * final_btc_price
        spot_pnl = spot_value - total_reinvested
        
        return {
            "total_reinvested_usd": total_reinvested,
            "spot_btc_accumulated": spot_btc,
            "avg_cost_basis": spot_cost_basis,
            "final_spot_value": spot_value,
            "spot_pnl_usd": spot_pnl,
            "spot_pnl_pct": (spot_pnl / total_reinvested * 100) if total_reinvested > 0 else 0,
            "combined_return_pct": "FA수익 + 현물수익 합산",
        }
```

### 1-3. ATR 기반 포지션 사이징

```python
def calculate_position_size(
    equity: float, entry_price: float, atr: float,
    risk_pct: float = 0.01, stop_atr_mult: float = 2.0,
    max_position_pct: float = 0.20, max_leverage: float = 2.0,
) -> float:
    risk_amount = equity * risk_pct
    stop_distance = atr * stop_atr_mult
    if stop_distance <= 0:
        return 0.0
    raw_size = risk_amount / stop_distance
    max_by_notional = (equity * max_position_pct) / entry_price
    max_by_leverage = (equity * max_leverage) / entry_price
    return min(raw_size, max_by_notional, max_by_leverage)
```

### 1-4. 거래 빈도 필터 (Test I 교훈 적용)

```python
MIN_TRADES_6Y = 200       # 6년 최소 200회 (월 2.8회)
MIN_TRADES_PER_WF = 10    # WF 윈도우(90일)당 최소 10회

def check_trade_frequency(total_trades: int, n_years: float = 6.0) -> str:
    """
    거래 빈도가 너무 낮으면 Walk-Forward 검증이 무의미.
    Test I에서 DCA rsi_trend_guard가 68회/6년으로 WF 실패한 전례.
    """
    if total_trades < MIN_TRADES_6Y:
        return f"⚠️ 탈락: {total_trades}회 < 최소 {MIN_TRADES_6Y}회 — WF 통계 불안정"
    avg_per_window = total_trades / (n_years * 365 / 90)
    if avg_per_window < MIN_TRADES_PER_WF:
        return f"⚠️ 경고: 윈도우당 {avg_per_window:.1f}회 < 최소 {MIN_TRADES_PER_WF}회"
    return f"✅ 통과: {total_trades}회, 윈도우당 {avg_per_window:.1f}회"
```

### 2-3. 목표 함수

```python
def objective(result) -> float:
    sharpe = result.sharpe_ratio
    annual_return = result.annualized_return_pct
    max_dd = result.max_drawdown_pct
    consistency = result.pct_positive_months / 100
    calmar = result.calmar_ratio
    
    # MDD 패널티 (20% 초과 시 강력 패널티)
    dd_penalty = max(0, (max_dd - 20) * 0.5) if max_dd > 20 else max(0, (max_dd - 10) * 0.2) if max_dd > 10 else 0
    
    score = (
        annual_return * 0.30 +
        sharpe * 5.0 * 0.25 +
        calmar * 2.0 * 0.20 +
        consistency * 10.0 * 0.15 +
        - dd_penalty * 0.10
    )
    return score
```

```python
class OnchainFilter:
    def should_enter_long(self, data: dict) -> tuple[bool, str]:
        checks = []
        
        # OI: 가격 상승 + OI 감소 5%+ = 숏 커버링, 지속성 낮음
        if data.get("oi_change_pct_1h", 0) < -5.0:
            checks.append((False, "OI 감소 — 숏커버 의심"))
        
        # 펀딩비 과열: > 0.1% = 롱 과열
        if data.get("funding_rate", 0) > 0.001:
            checks.append((False, "펀딩비 과열"))
        
        # 롱 비율 과열: > 70%
        if data.get("long_ratio", 50) > 70:
            checks.append((False, "롱 비율 과열"))
        
        failed = [c for c in checks if not c[0]]
        return len(failed) == 0, "; ".join(f[1] for f in failed) or "OK"
    
    def should_enter_short(self, data: dict) -> tuple[bool, str]:
        checks = []
        
        if data.get("oi_change_pct_1h", 0) < -5.0:
            checks.append((False, "OI 감소"))
        if data.get("funding_rate", 0) < -0.0005:
            checks.append((False, "펀딩비 숏 과열"))
        if (100 - data.get("long_ratio", 50)) > 60:
            checks.append((False, "숏 비율 과열"))
        
        failed = [c for c in checks if not c[0]]
        return len(failed) == 0, "; ".join(f[1] for f in failed) or "OK"
```

## 만약 추세추종도 Walk-Forward를 통과하지 못한다면

DCA처럼 추세추종도 WF 실패할 가능성이 있다. 그 경우의 대안:

```
[대안 1] FA 단독 + 수익 재투자 (Stage 0 결과가 양호한 경우)
  → FA 수익의 30%를 BTC 현물 매수
  → 원금 리스크 0, 장기 BTC 축적
  → 이 조합은 WF 검증 불필요 (FA 자체가 이미 WF 통과)

[대안 2] FA 자본 비중 확대
  → 현재 FA 50% + Cash 50%
  → FA 70% + Cash 30%으로 변경
  → FA 단독 Sharpe 1.494 (6년) → 자본 확대로 절대 수익 증가
  → MDD 0.94% → ~1.3%로 소폭 증가, 여전히 안전 범위

[대안 3] 멀티 심볼 FA
  → 현재 BTCUSDT 단일 → ETHUSDT, SOLUSDT 추가
  → 각 심볼의 펀딩비 기회를 독립적으로 포착
  → 거래 빈도 증가 → 연 수익률 상승 가능

어떤 경우든 "방향성 베팅이 검증 실패 시 무리하게 적용하지 않는다"가 원칙이다.
```

## 주의사항

1. **거래 빈도 최우선 확인**: 6년 200회 미만 변형은 즉시 탈락. Test I의 교훈.
2. **롱+숏 양방향 필수**: DCA는 롱 전용이라 실패. 추세추종은 2022년 숏 수익이 핵심.
3. **수수료·슬리피지 반드시 포함**: 추세추종은 DCA보다 거래 빈도 높음 → 수수료 영향 큼.
4. **BTC Buy & Hold 벤치마크 필수**: 추세추종이 단순 보유보다 나아야 의미 있음.
5. **레버리지 2배 제한 유지**: 프로젝트 정책 불변.
6. **기존 FA 코드 수정 금지**: TF는 별도 모듈로 추가.
7. **Stage 0은 병렬 실행**: 추세추종 결과와 무관하게 FA 수익 재투자는 독립 검증.
