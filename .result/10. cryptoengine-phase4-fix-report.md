## 2. 이번 작업에서 수정된 항목

### A1: strategy_id 불일치 수정

**파일**: `cryptoengine/services/strategies/funding-arb/main.py`

**문제**: 오케스트레이터는 `strategy:command:funding-arb` 채널로 발행하지만, funding-arb 서비스의 기본 strategy_id가 `"funding-arb-01"`로 설정되어 `strategy:command:funding-arb-01`을 구독 → 명령 수신 불가.

**수정**:
```python
# Before
strategy_id = os.getenv("STRATEGY_ID", "funding-arb-01")
# After
strategy_id = os.getenv("STRATEGY_ID", "funding-arb")
```

### A4: DB 스키마 불일치 4건 전면 수정

#### A4-1: ohlcv_history 컬럼명 (`ts` → `timestamp`)

**파일**: `cryptoengine/services/market-data/collector.py`

**문제**: `INSERT INTO ohlcv_history (..., ts, ...)` + `ON CONFLICT (..., ts)` — 스키마는 `timestamp` 컬럼 사용.

**수정**: 모든 `ts` → `timestamp` 변경.

#### A4-2: market_regimes → market_regime_history

**파일**: `cryptoengine/services/market-data/regime_detector.py`

**문제**: 테이블명 `market_regimes` (존재하지 않음) + 평면 컬럼 `(regime, confidence, adx, volatility, bb_width)` — 스키마는 `market_regime_history` 테이블에 `indicators JSONB` 컬럼 사용.

**수정**:
```sql
-- Before
INSERT INTO market_regimes (regime, confidence, adx, volatility, bb_width)
-- After
INSERT INTO market_regime_history (symbol, regime, confidence, indicators)
```
`adx`, `volatility`, `bb_width`는 `{"adx": ..., "volatility": ..., "bb_width": ...}` JSONB로 패킹.

**파일**: `cryptoengine/services/market-data/main.py`  
런타임 DDL도 `market_regime_history` 정확한 스키마로 업데이트.

#### A4-3: funding_rates → funding_rate_history

**파일 1**: `cryptoengine/services/market-data/funding_monitor.py`  
**파일 2**: `cryptoengine/shared/db/repository.py` (FundingRepo)

**문제**: 테이블명 `funding_rates` (존재하지 않음) + `next_funding_time` 컬럼 — 스키마는 `funding_rate_history` 테이블에 `timestamp` 컬럼 사용.

**수정**:
```sql
-- Before (funding_monitor.py)
INSERT INTO funding_rates (exchange, symbol, rate, predicted_rate, next_funding_time)
ON CONFLICT (exchange, symbol, next_funding_time)
-- After
INSERT INTO funding_rate_history (exchange, symbol, rate, predicted_rate, timestamp)
ON CONFLICT (exchange, symbol, timestamp)
```

`shared/db/repository.py`에서도 `id` 파라미터 제거 (BIGSERIAL 자동생성), `collected_at` → `timestamp` 변경, SELECT도 `funding_rate_history`로 변경.

#### A4-4: portfolio_snapshots 스키마 동기화

**파일**: `cryptoengine/shared/db/init_schema.sql`

**문제**: `init_schema.sql`의 `portfolio_snapshots`가 `realized_pnl` (4개 컬럼)으로 정의되어 있어, `portfolio_monitor.py`가 런타임에 `CREATE TABLE IF NOT EXISTS` (no-op)을 실행한 후 `realized_pnl_today` 컬럼 INSERT 시 런타임 오류 발생.

**수정**: `init_schema.sql`의 `portfolio_snapshots`를 `portfolio_monitor.py`가 사용하는 실제 스키마로 업데이트:
```sql
realized_pnl_today DOUBLE PRECISION DEFAULT 0,
daily_drawdown DOUBLE PRECISION DEFAULT 0,
weekly_drawdown DOUBLE PRECISION DEFAULT 0,
monthly_drawdown DOUBLE PRECISION DEFAULT 0,
sharpe_ratio_30d DOUBLE PRECISION,
strategies JSONB DEFAULT '[]',
snapshot_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
```

**파일**: `cryptoengine/shared/db/repository.py` (SnapshotRepo)  
`id` 제거, `created_at` → `snapshot_at`, `kill_switch_triggered` SQL 컬럼 제거.

**파일**: Grafana 대시보드 JSON (`strategy_monitor.json`, `live_performance.json`)  
쿼리에서 `realized_pnl` → `realized_pnl_today` 변경.

### B5: SafetyGuard record_api_response() 연결

**파일**: `cryptoengine/services/execution/engine.py`

**문제**: SafetyGuard가 "마지막 API 응답 후 30초 초과 시 주문 차단" 로직을 갖고 있지만, 성공적인 주문 후 `record_api_response()`가 호출되지 않아 타이머가 리셋되지 않음 → 30초 이후 모든 주문 차단.

**수정**: `_process_order()` 내 성공적인 `place_order()` 호출 직후에 추가:
```python
self._safety.record_api_response()
self._safety.record_api_call()
```

## 4. 통합 검증 체크리스트

수정 완료 후 아래 순서로 검증 권장:

```bash
# 1. 전체 재빌드 (shared/ 변경 포함)
docker compose build --no-cache market-data execution-engine funding-arb strategy-orchestrator

# 2. 인프라 기동
docker compose up -d postgres redis

# 3. 핵심 서비스 기동
docker compose up -d market-data execution-engine funding-arb strategy-orchestrator

# 4. 30초 대기 후 상태 확인
sleep 30 && docker compose ps
# 모든 서비스 Running 상태 (Restarting 없어야 함)

# 5. ohlcv 데이터 수집 확인
docker compose exec postgres psql -U cryptoengine -d cryptoengine \
  -c "SELECT count(*) FROM ohlcv_history;"
# count > 0 이면 성공

# 6. 레짐 데이터 확인
docker compose exec postgres psql -U cryptoengine -d cryptoengine \
  -c "SELECT count(*) FROM market_regime_history;"

# 7. 펀딩비 데이터 확인
docker compose exec postgres psql -U cryptoengine -d cryptoengine \
  -c "SELECT count(*) FROM funding_rate_history;"

# 8. 오케스트레이터 → FA 명령 전달 확인
docker compose logs --tail=20 funding-arb | grep -i "command\|allocation\|start"
# "Received command" 또는 "allocation_received" 로그가 보이면 성공

# 9. 레버리지/마진 설정 확인
docker compose logs funding-arb | grep -i "leverage_set\|margin_mode_set"

# 10. 주문이 실행엔진을 경유하는지 확인
docker compose logs execution-engine | grep -i "order_received\|safety_checks_passed"
```
