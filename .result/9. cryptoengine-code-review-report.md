## 1. 전체 요약

### **현재 성숙도: Level 3 (베타)**

> 리스크 관리 구조가 존재하고, 백테스팅이 부분적으로 구현되어 있으나, 핵심 실행 안전장치 3개(레버리지 설정, 마진 모드, 주문 라우팅)가 누락된 치명적 결함이 있다. 모니터링과 인프라는 양호하다.

```
Level 1 프로토타입  ──────────────────────── ← 기초만 존재
Level 2 알파       ──────────────────────── ← 핵심 기능 존재, 리스크 미흡
Level 3 베타  ◀━━ 현재 위치 ━━━━━━━━━━━━━━━← 리스크 구조 존재, 실행 결함
Level 4 프로덕션   ──────────────────────── ← 대부분 충족
Level 5 프로덕션안정 ─────────────────────── ← 전 영역 충족
```

## 3. 치명적 결함 목록 🔴

> **Phase 5(실전) 전에 반드시 해결해야 하는 문제들**

### [치명 1] `set_leverage()` 미호출 — 거래소 기본값 레버리지로 주문

**위치**: `shared/exchange/bybit.py`, `services/execution/order_manager.py`

`ExchangeConnector` ABC에 `set_leverage()` 추상 메서드가 없고, `BybitConnector.place_order()`도, `FundingArbStrategy.on_start()`도 레버리지 설정 API를 호출하지 않는다. Bybit 기본 레버리지는 최대 100배까지 될 수 있으며, **현재 시스템은 미확인 레버리지로 주문을 제출하고 있다.**

### [치명 2] `set_margin_mode('isolated')` 미호출 — 교차마진 위험

**위치**: `shared/exchange/bybit.py`, `services/execution/order_manager.py`

교차마진(Cross Margin)이 기본값인 경우, 선물 포지션 손실이 전체 계좌를 청산시킬 수 있다. 격리마진(Isolated) 설정 API가 어디에도 호출되지 않는다.

### [치명 3] FA 전략이 실행엔진을 우회하여 직접 주문

**위치**: `services/strategies/funding-arb/strategy.py:546-558`

`FundingArbStrategy._place_and_track()`이 `self._exchange.place_order(order)`를 직접 호출한다. `SafetyGuard`의 6단계 안전 검증(레버리지, 마진, 슬리피지, 네트워크, 레이트 제한)을 완전히 우회한다. **실행 안전 아키텍처의 핵심 위반.**

### [치명 4] Redis 채널명 불일치 — 전략이 오케스트레이터 명령을 수신 못함

**위치**: `services/orchestrator/core.py:33-37` vs `services/strategies/base_strategy.py`

오케스트레이터는 `strategy:funding_arb:command`로 발행하지만, 전략은 `strategy:command:funding-arb`를 구독한다. **오케스트레이터의 모든 전략 명령(자본 배분, 시작/정지)이 유실된다.**

### [치명 5] 두 개의 Kill Switch 구현이 통합되지 않음

**위치**: `shared/kill_switch.py` vs `services/orchestrator/core.py`

`shared/kill_switch.py` (일일 -1% 임계값)와 `orchestrator/core.py`의 자체 `KillSwitchState` (일일 -5% 임계값)가 병렬 존재. `shared` 버전은 오케스트레이터에서 import되지 않으며, 어느 쪽이 실제로 동작하는지 불명확하다.

### [치명 6] 스키마 불일치 4개 — DB 런타임 오류

**위치**: `services/market-data/collector.py`, `services/orchestrator/repository.py`

| 코드에서 사용 | 스키마 정의 |
|------------|-----------|
| `INSERT INTO ohlcv` | `ohlcv_history` |
| `INSERT INTO market_regimes` | `market_regime_history` |
| `INSERT INTO funding_rates` | `funding_rate_history` |
| `realized_pnl_today` 컬럼 | `realized_pnl` |

→ market-data, orchestrator 서비스가 실행 중이더라도 **DB 쓰기가 전부 실패하고 있을 가능성이 높다.**

## 6. 전설적 트레이더 원칙 준수 체크리스트

| # | 원칙 | 구현 | 구현 방식 | 개선 필요 |
|---|------|------|---------|----------|
| 1 | **리스크 관리 > 적중률** | ⚠️ 부분 | Kill Switch 4단계 존재, SafetyGuard 존재 | 거래당 1-2% 리스크 한도 미구현, ATR 포지션 사이징 없음 |
| 2 | **낮은 레버리지 (2~5배)** | ❌ 미구현 | FA 기본값 1.0x (설정), 그러나 exchange에 `set_leverage()` 미호출 → 실제 적용값 미확인 | `set_leverage()` 즉시 구현 필수 |
| 3 | **역발상 사고 (센티먼트 역행)** | ⚠️ 부분 | DCA에 Fear&Greed 통합, FA config에 `funding_rate > 0.05%` 역발상 명시 | 코드에 미구현 (config만 존재), OI 다이버전스 없음 |
| 4 | **감정 제거 (체계적 규칙)** | ✅ 양호 | 상태머신 기반 FA, 레짐 기반 가중치, 자동 Kill Switch | — |

## 8. 긍정적 평가 — 잘 만들어진 부분

1. **`ExchangeConnector` ABC + factory 패턴**: 새 거래소 추가 시 하나의 클래스만 구현하면 됨. 설계 우수.
2. **`BaseStrategy` ABC**: 전략 간 코드 공유, Redis 기반 통신, 상태 관리 잘 구조화됨.
3. **6년 WFO 백테스팅 인프라**: 12개 전문 테스트 스크립트, 파라미터 스윕, 스트레스 테스트까지 구축. 업계 수준.
4. **WebSocket 재연결 로직**: 지수 백오프, ping/pong, 최대 120초 제한 등 프로덕션급 구현.
5. **Delta-Neutral 로직**: `DeltaNeutralManager`의 스프레드 모니터링, 마진 비율 계산은 FA 전략에 적합.
6. **Telegram 봇**: 긴급 청산, 전략 개별 제어, 실시간 상태 조회 완비.
7. **레짐 기반 동적 가중치**: `WeightManager` EMA 스무딩 + 4개 레짐 × 전략별 가중치 = 백테스트 검증된 설계.
